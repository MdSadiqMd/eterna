### Key findings

**`HalfBook::consume` is now a named, visible frame (30%)** ‚Äî the best-pointer update and level removal that were previously scattered inline inside `submit` are now isolated. This is useful: profiling can now target `consume` independently.

**`BTreeMap::remove` dropped out of the top-level frame** ‚Äî it's still called (inside `consume`) but no longer a direct child of `submit`. The O(log n) cost is still there but it's owned by a named function.

**Dealloc rose from 14% ‚Üí 20%** ‚Äî the bench workload heavily exhausts levels (aggressive sweeps). The `VecDeque` backing buffer free on level drain remains the dominant allocator cost. With the refactor, each `consume` call is leaner, making the allocator a relatively larger fraction.

**`VecDeque::grow` dropped from 2 ‚Üí 1 sample** ‚Äî fewer initial allocations because `HalfBook::rest` reuses existing price levels more efficiently.

### Remaining bottlenecks

1. **Allocator (20-25% combined)** ‚Äî `dealloc` + `grow_one` dominate after matching. Every level exhaustion frees a `VecDeque` buffer; every new level allocates one. This is the primary remaining cost.

2. **`BTreeMap::remove` inside `consume`** ‚Äî still O(log n) per exhausted level. Unavoidable with BTreeMap; a fixed price-ladder array would make this O(1).

### Next optimisations

1. **Pool `VecDeque` allocations** ‚Äî keep a free-list of drained `VecDeque<Order>` buffers and reuse them when a new order arrives at a previously-empty price. Eliminates the dealloc/alloc round-trip entirely.

2. **Price ladder array** ‚Äî replace `BTreeMap<u64, VecDeque<Order>>` with `[Option<VecDeque<Order>>; MAX_TICKS]` for O(1) access if the tick range is bounded.

3. **Pre-size `VecDeque` with `with_capacity(4)`** on first insert at a new level ‚Äî removes `grow_one` reallocs for the common case of a few orders per level.
